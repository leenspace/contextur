Deep-scan this repository with reviewer-focused subagents and personalize the Contextur reviewers so code review catches repo-specific violations, not just generic ones.

Contextur itself does not call an LLM. You are the agentic development tool that runs this protocol. Use your available subagent/task mechanism when the tool supports it. If the tool does not support true subagents, run the same reviewer research tasks sequentially and keep their outputs separated by reviewer id.

## Step 1 - Ingest authoritative context first

Before inferring anything new, read any of these files that exist. They often contain institutional knowledge the team has already written down. Treat them as authoritative unless code evidence clearly contradicts them.

- `.contextur/repo-snapshot.yaml` (generated by `contextur init`; use this as the baseline repo map)
- `.contextur/manifest.yaml` (defines reviewer ids, prompt paths, triggers, and mandatory reviewers)
- `.contextur/reviewers/*.md` (read the generated reviewer scopes before assigning research)
- `AGENTS.md` (any AI assistant)
- `.cursorrules`, `.cursor/rules/*.mdc` (Cursor)
- `CLAUDE.md`, `.claude/commands/*.md` (Claude Code)
- `.github/copilot-instructions.md`, `copilot-instructions.md` (Copilot)
- `.aider.conf.yml`, `.windsurfrules` (other AI editors)
- `ARCHITECTURE.md`, `CONTRIBUTING.md`, `docs/architecture/**`, `docs/**/ARCHITECTURE.md`
- `README.md` and the top-level `docs/` tree

Do not duplicate or rewrite rules that already exist in these files. Reference them by path and section heading when you cite them in reviewer rules.
If `.contextur/repo-snapshot.yaml` conflicts with newer code evidence, prefer code evidence and mention the mismatch in your summary.

## Step 2 - Build the shared research packet

Prepare one shared packet for all research subagents:

- Existing authoritative docs found in Step 1, with paths and short notes.
- Parsed reviewer list from `.contextur/manifest.yaml`.
- All package manifests: `package.json`, `pyproject.toml`, `requirements.txt`, `go.mod`, `Cargo.toml`, `pubspec.yaml`, `Gemfile`, `composer.json`, `pom.xml`, `build.gradle`, `build.gradle.kts`.
- Top-level directory tree, 1-2 levels deep.
- Representative source files across layers: entry points, API/routes, services/domain, data access, config, one or two tests, and any files named by architecture docs.
- Known commands from package manifests and `.contextur/repo-snapshot.yaml`.

Keep the packet factual. It should contain evidence and paths, not conclusions.

## Step 3 - Launch reviewer research subagents in parallel

Launch all applicable reviewer research subagents in a single parallel batch. Mandatory baseline reviewers always run:

- `correctness`
- `security`
- `architecture`
- `testing`
- `operability`

Also launch optional reviewer research subagents when `.contextur/manifest.yaml`, `.contextur/repo-snapshot.yaml`, or repo evidence shows they are relevant:

- `performance`
- `api-contract`
- `data-migration`
- `ci-release`
- `maintainability`
- Any existing custom reviewer in `.contextur/reviewers/`

Pass the shared research packet to every subagent. Each subagent must answer only for its reviewer scope and must not edit files.

## Reviewer-question matrix

Use these questions as the prompt for each reviewer research subagent.

### correctness

Ask:

- What domain invariants, lifecycle rules, state transitions, and async/concurrency patterns must never regress?
- Where does the repo parse, map, validate, normalize, or default external data, and what failure modes are common?
- Which error-handling conventions are real in code: thrown errors, result types, retries, rollbacks, fallback values, or state emissions?
- Which directories or symbols are recent hotspots for logic bugs or high churn?

Return candidate rules only when they cite a concrete path, symbol, or authoritative doc section.

### security

Ask:

- Where are authentication, authorization, sessions, tenant/user boundaries, and role checks implemented?
- Where does input validation happen for HTTP, RPC, CLI, background jobs, webhooks, files, or queue payloads?
- Where are secrets, tokens, credentials, encryption keys, and sensitive personal data loaded, logged, stored, or returned?
- Which external services, SSRF-capable calls, redirects, deserialization paths, or command/query construction points exist?

Return rules as enforceable threat-model checks with concrete exploit-relevant paths.

### architecture

Ask:

- What architectural pattern is actually used: layered, clean, hexagonal, feature-first, package-per-domain, microservice, plugin, or another pattern?
- What are the dependency direction rules between UI, domain, data, infrastructure, shared packages, and generated code?
- Which public API surfaces, barrels, exported modules, route registries, dependency-injection roots, or package boundaries are stable contracts?
- Which direct imports, re-exports, framework leaks, or cross-package dependencies should reviewers flag?

Return rules that state the allowed boundary and the evidence path that proves it.

### testing

Ask:

- Which test frameworks, file naming conventions, fixture patterns, mocks/fakes, snapshots/goldens, and contract tests are used?
- Which commands should reviewers request for unit, integration, e2e, typecheck, lint, build, or generated-code validation?
- Where are tests colocated or centralized, and which source changes normally require test changes?
- What makes tests flaky in this repo: timers, network, clock, database, shared state, generated files, or parallelism?

Return rules that connect changed behavior to the expected test evidence.

### operability

Ask:

- Where are logs, metrics, traces, health checks, diagnostics, runbooks, alerts, and dashboards wired?
- How does the repo handle retries, timeouts, backoff, circuit breakers, idempotency, queues, schedulers, and background jobs?
- Where are configuration, feature flags, rollout controls, environment defaults, and deployment assumptions defined?
- Which failure modes would be silent, noisy, expensive, hard to roll back, or hard for on-call engineers to diagnose?

Return rules that tie code patterns to concrete production or support impact.

### performance

Run when code, dependencies, or manifest triggers indicate runtime performance matters.

Ask:

- What are the hot paths, large collections, render loops, request fan-out, cache layers, database access patterns, or expensive serialization paths?
- Which batching, pagination, memoization, indexing, streaming, lazy-loading, or resource-lifecycle conventions are already used?
- Which performance tests, benchmarks, profiling tools, budgets, or bundle-size checks exist?

### api-contract

Run when the repo has REST/OpenAPI, GraphQL, RPC, proto, event schemas, SDKs, generated clients, or public route contracts.

Ask:

- Where are external contracts defined and generated?
- What compatibility rules exist for request/response fields, status codes, errors, events, versioning, pagination, and auth semantics?
- Which consumers, SDKs, or downstream services depend on these contracts?

### data-migration

Run when the repo has migrations, schema files, ORM models, SQL, backfills, or persisted data contracts.

Ask:

- Where are schema changes, migrations, seed data, backfills, indexes, constraints, and rollbacks defined?
- What deploy-order, backward/forward compatibility, locking, data-loss, and large-table rules are visible?
- Which commands or environments validate migrations?

### ci-release

Run when the repo has CI, containers, deploy config, release scripts, package publishing, or infrastructure automation.

Ask:

- Which workflows build, test, package, deploy, publish, migrate, or roll back the system?
- What cache, artifact, secret, environment, permission, provenance, and promotion rules should reviewers enforce?
- Which changes require CI/release reviewer attention even if app code is untouched?

### maintainability

Run for code-heavy repos and monorepos.

Ask:

- Which abstractions, module boundaries, naming conventions, generated-code boundaries, and ownership patterns keep changes maintainable?
- Where are duplication, oversized modules, god services, cyclic dependencies, dead code, or brittle conventions likely?
- Which refactors are appropriate to suggest during review versus too broad for a PR?

## Required subagent return format

Each subagent must return Markdown with these exact headings:

```markdown
## reviewer_id
<reviewer id>

## evidence_paths
- `<path>` - what this file/doc proves

## candidate_rules
- [reviewer_id] Concrete enforceable rule citing `<path>` or `symbolName`. Explain what reviewers should flag.

## suggested_specialist_reviewers
- Optional reviewer id, trigger glob, and why it is distinct from the baseline reviewers. Use `none` if not needed.

## uncertainties
- Facts that need human confirmation or more code evidence.

## do_not_infer
- Plausible but unsupported rules the synthesizer must not add.
```

## Step 4 - Synthesize repo-specific rules

After all subagents finish:

1. Deduplicate overlapping candidate rules.
2. Drop any rule that lacks a concrete path, symbol, manifest trigger, or authoritative doc section.
3. Drop style-only preferences unless they are already documented in authoritative repo docs.
4. Keep rules scoped to the reviewer that can enforce them during code review.
5. Prefer 3-7 high-signal rules per baseline reviewer over long generic lists.

For each reviewer file that receives rules, append this section at the end if it does not exist:

```markdown
## Repo-specific rules

<!-- contextur:repo-specific-start -->
- <concrete rule>. Example: "All database queries go through `src/db/client.ts`; direct imports of `@prisma/client` outside that file are a violation."
- <concrete rule>. Every rule MUST cite a concrete path, symbol, or authoritative doc section in this repo.
- <concrete rule>. If the rule is already documented elsewhere, reference the source (for example, "See `docs/ARCHITECTURE.md` section `State Management`").
<!-- contextur:repo-specific-end -->
```

If the section already exists from a prior run, replace only the content between the `<!-- contextur:repo-specific-start -->` and `<!-- contextur:repo-specific-end -->` markers. Do not append a duplicate block. Do not touch generated reviewer text outside the markers.

The markers let `contextur-update` find and replace repo-specific rules later without touching the rest of the file.

## Step 5 - Add specialized reviewers conservatively

Create or enable a specialist reviewer only when the research output shows all of these:

- A distinct recurring subsystem or contract not covered by the baseline reviewers.
- Concrete files or globs that can trigger the reviewer.
- At least three enforceable rules backed by paths or authoritative docs.

Examples:

- A Flutter app with BLoC -> `.contextur/reviewers/flutter-bloc.md`
- A GraphQL schema -> `.contextur/reviewers/graphql-schema.md`
- Terraform infra -> `.contextur/reviewers/terraform.md`
- A specific API auth flow -> `.contextur/reviewers/api-auth.md`

Use the same structure as the existing base reviewers. Then add a corresponding entry to `.contextur/manifest.yaml` with an appropriate `trigger` glob and `mandatory: false`:

```yaml
  - id: flutter-bloc
    path: reviewers/flutter-bloc.md
    trigger: "**/*_bloc.dart"
    mandatory: false
```

## Step 6 - Update AGENTS.md

Read the existing `AGENTS.md` first. Preserve any hand-written sections. Replace only the `<!-- contextur:inferred -->` block with a concrete `## Architecture` section covering:

- Tech stack: languages, major frameworks, and versions.
- Architecture and directory layout.
- Key conventions: naming, testing, where to find things, and which reviewer rules are authoritative.

## Step 7 - Summarize

At the end, tell the user:

1. Which existing AI-friendly docs you found and leveraged.
2. Which research subagents ran and which optional ones were skipped.
3. Which files you modified, one-line summary each.
4. Any new reviewer files you created.
5. Anything you were unsure about, so the user can decide whether to keep, refine, or delete it.
