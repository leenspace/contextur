Re-scan this repository and update the Contextur reviewers to reflect recent changes. This is idempotent — re-running should not produce duplicates or drift.

## Step 1 — Read the current state

Read, in this order:

1. **Current repo-specific rules** in each mandatory baseline reviewer file. Prefer `.contextur/reviewers/correctness.md`, `security.md`, `architecture.md`, `testing.md`, and `operability.md`; also read legacy `.contextur/reviewers/core-logic.md` if present. Look for the `<!-- contextur:repo-specific-start -->` / `<!-- contextur:repo-specific-end -->` markers. This tells you what has already been documented.
2. **Any custom/optional reviewers** in `.contextur/reviewers/` beyond the mandatory baseline, plus `.contextur/manifest.yaml`.
3. **Generated repo snapshot** in `.contextur/repo-snapshot.yaml` (treat as baseline unless contradicted by newer code evidence).
4. **Existing AI-friendly docs** (same list as init):

   - `ARCHITECTURE.md`, `CONTRIBUTING.md`, `docs/architecture/**`, `README.md`

## Step 2 — Detect what has evolved

- Recent commits: `git log --oneline -50`
- Recently-changed files: `git diff HEAD~20 --stat` (or `HEAD~50` for larger sweeps)
- New directories that didn't exist before (compare against paths cited in current reviewer rules)
- New dependencies in package manifests

## Step 3 — Identify drift

For each rule currently inside a `<!-- contextur:repo-specific-start -->` block:

- **Stale**: cites a path that no longer exists, or a pattern that's been refactored away.
- **Still valid, wording outdated**: rule is correct but references an old file name or framework version.
- **Incomplete**: a new convention has been adopted (e.g., a new state-management lib, a new API layer) that isn't reflected.

Also cross-check with external docs: if `ARCHITECTURE.md` has changed in ways that contradict a current repo-specific rule, flag it.

## Step 4 — Rewrite the repo-specific blocks

For each mandatory baseline reviewer file present (`correctness.md`, `security.md`, `architecture.md`, `testing.md`, `operability.md`; or legacy `core-logic.md`):
1. Locate the `<!-- contextur:repo-specific-start -->` / `<!-- contextur:repo-specific-end -->` markers.
2. **Replace** the content between the markers with the updated rules.
3. Do NOT append a new block. Do NOT touch anything outside the markers.

If the markers don't exist (reviewer was never personalized), create a `## Repo-specific rules` section at the end using the same format as `/project:contextur-init`.

## Step 5 — Review custom reviewer files

For each custom or optional reviewer in `.contextur/reviewers/` (beyond the mandatory baseline):

- If it references paths that no longer exist: flag it to the user, propose a fix, and ask before rewriting.
- If a new specialized area has emerged (e.g., a new subsystem was added since last update) that warrants its own reviewer: propose creating it.

## Step 6 — Update AGENTS.md

Re-read AGENTS.md. If the `## Architecture` section (originally generated by `/project:contextur-init`) is out of date, update it. Preserve all hand-written sections.

## Step 7 — Summarize

Tell the user:
1. Which rules were **updated** (stale → new wording).
2. Which rules were **removed** (no longer apply).
3. Which rules were **added** (new convention detected).
4. Which custom reviewers look stale and need attention.
5. Anything you couldn't decide about — leave it for the user.

Before writing any changes, show a short diff-style summary and ask the user to confirm.
